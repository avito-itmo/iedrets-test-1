# Курсовой проект по системному проектированию

Этот репозиторий используется для выполнения курсового проекта,
состоящего из нескольких лабораторных работ (ЛР).\
Итогом семестра является **единый архитектурный документ** в формате
Markdown.

Проект развивается поэтапно: каждая лабораторная работа дополняет и
уточняет архитектуру одной и той же системы.

# 📦 Структура репозитория

    architecture.md   # основной и единственный итоговый документ
    assets/           # изображения (png/svg/jpg)

Дополнительные файлы допускаются только при обоснованной необходимости.

# 📄 Требования к документу

## 1. Формат

-   Используется **Markdown**.
-   Поддерживаемый диалект - [GitHub Flavored Markdown (GFM)](https://docs.github.com/ru/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax).
-   Документ должен корректно отображаться в веб-интерфейсе GitHub.
-   В репозитории хранится только текст и связанные с ним ресурсы.

## 2. Диаграммы

-   Основной формат диаграмм - `mermaid`.
-   Диаграммы вставляются непосредственно в `architecture.md`.

Пример:

    ``` mermaid
    graph TD
        A[Client] --> B[API]
    ```

Требования:

-   Диаграммы должны корректно рендериться в GitHub.
-   Диаграммы должны быть читаемыми.
-   Не допускаются синтаксические ошибки.

## 3. Изображения

Если необходимо вставить изображение:

-   Файл размещается в каталоге `assets/`.
-   Используется относительная ссылка.

Пример:

    ![Контекстная диаграмма](assets/context.png)


## 4. Ссылки

-   **Запрещены ссылки на внешние ресурсы.**
-   Вся необходимая информация должна находиться внутри репозитория.
-   Допускаются только относительные ссылки на файлы внутри проекта.

Документ должен быть полностью самодостаточным.

# 🔀 Правила работы с Git

## 1. Основная ветка

-   Прямые push в `main` запрещены.
-   Работа ведётся только через Pull Request.

## 2. Правило «один PR - одна лабораторная»

Каждая лабораторная работа:

-   выполняется в отдельной ветке (`lab-1`, `lab-2`, ...);
-   оформляется отдельным Pull Request;
-   не объединяется с другими лабораторными.

## 3. Merge после проверки

После получения approval:

-   Pull Request обязательно мёржится в `main`;
-   следующая лабораторная начинается от актуального `main`.

# 🧱 Эволюция архитектуры

Проект является постепенным развитием одной архитектуры.

В новой лабораторной:

-   запрещено радикально переписывать результат предыдущей ЛР;
-   допускается уточнение и развитие решений;
-   существенные изменения должны быть явно объяснены.

Если архитектурное решение изменяется:

-   должно быть описано, что именно изменилось;
-   должны быть указаны причины изменения;
-   должны быть описаны последствия для системы.


Все вопросы и уточнения оформляются через Issues в данном репозитории.
